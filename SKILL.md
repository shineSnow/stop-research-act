# Stop-Research-Act: 问题解决 SOP

## Overview

大模型在执行任务时天然倾向于"失败 → 立即换方案重试"——这是一个严重的系统性缺陷。每一次失败都包含诊断信息，盲目重试不仅浪费时间，还会引入新变量（残留进程、配置冲突、环境破坏），让问题越来越复杂。

**核心原则：遇到任何失败，先停下来理解为什么失败，再行动。**

## The Iron Law

```
失败后，禁止不诊断就直接换方案重试。
不确定根因时，禁止凭猜测行动。
```

## When to Use

这个 SOP 适用于**任何任务执行过程中遇到失败或意外结果**：

- 命令执行报错（exit code != 0）
- 工具返回意外的结果
- 页面/服务状态与预期不符
- 连接失败、超时、权限拒绝
- 配置不生效
- **尤其是：已经尝试了 2 次以上还没解决**

**不需要使用的情况：**
- 拼写错误、路径错误等一目了然的低级失误（修正即可）
- 用户明确给出了修正指令

## 五步闭环

```
执行任务 → 遇到失败 → [STOP] 理解失败 → [RESEARCH] 搜索研究 → [REFLECT] 形成假设 → [ACT] 精确修正 → 验证 → 回到执行
```

每一步都有明确的前置条件和输出要求。

---

### Step 1: [STOP] 停止并理解失败

**触发条件：** 任何操作失败、报错、返回意外结果。

**铁律：禁止在未完成这一步之前执行任何修复操作。**

**强制行为：**

1. **完整读取错误信息。** 不要只读第一行。错误信息、堆栈、状态码、返回体全部读完。
2. **提取关键线索。** 从错误中识别：什么操作失败了？在哪个环节（连接/认证/解析/超时/权限）？有哪些关键标识符（端口号、文件路径、状态码、错误码）？
3. **分类问题类型：**
   - A 类：**用法错误** — 参数不对、顺序不对、前提条件不满足
   - B 类：**环境/依赖问题** — 服务没启、端口冲突、版本不兼容、权限不足、残留进程
   - C 类：**目标系统行为变化** — API 改版、安全策略变更、协议升级、bug
4. **逐层验证依赖链。** 从最底层开始，用最原始的方式（curl、ls、cat、ping）验证每层依赖的状态。验证是读取状态，不是改变状态。

**输出模板：**
```
【诊断结果】
- 操作：[我做了什么]
- 错误：[完整错误信息]
- 关键线索：[具体标识符]
- 分类：[A/B/C]
- 已验证的依赖：[逐层状态]
- 怀疑点：[最可能的根因]
```

**如果无法确定根因或分类，进入 Step 2。**

---

### Step 2: [RESEARCH] 搜索研究

**触发条件：** Step 1 结束后，根因不确定，或确定是 A 类问题但不知道怎么正确使用。

**铁律：禁止凭猜测行动。必须先查文档/搜索。**

**强制行为（按优先级）：**

1. **查官方文档。** User guide、README、官方文档站点。这个工具/API 的正确用法是什么？我的用法和文档一致吗？
2. **搜索错误信息。** 把关键错误信息去敏后放入搜索。社区是否有相同问题？GitHub Issues 是否有相关讨论？
3. **查版本变更。** 如果工具/服务最近更新过，检查 changelog/release notes。行为变化往往在这里。
4. **搜索替代方案。** 如果是已知限制或 bug，官方或社区的 workaround 是什么？

**搜索原则：**
- 优先官方文档 > GitHub Issues > 社区博客 > StackOverflow
- 如果搜索结果不相关，说明搜索词有问题，换个角度搜
- 搜索结果不一定完全匹配你的场景，但可以提供方向
- 评估来源可信度：官方 > 维护者 > 高 star 项目 > 社区讨论

**输出模板：**
```
【研究结果】
- 查阅来源：[URL 或文档名称]
- 相关结论：[引用的关键信息]
- 适用性评估：[这个结论是否适用于我的场景]
```

---

### Step 3: [REFLECT] 反思与形成假设

**触发条件：** 研究完成后。

**强制行为：**

1. **连接信息。** 我的错误 + 研究结果 = 什么根因？
2. **如果之前已经失败过多次：** 每次失败的共同路径是什么？（同一个端口？同一个文件？同一个协议？）连续多个工具在同一个环节失败 → 问题在共同依赖上，不在任何一个工具上。
3. **形成可验证的假设：**
   - "我认为根因是 [XXX]"
   - "如果我做 [YYY]，应该看到 [ZZZ]"
   - "如果我的假设正确，修复后 [预期结果]"
   - "如果假设错误，我会看到 [什么信号]，然后我需要重新检查 [什么]"

**输出模板：**
```
【假设】
- 根因假设：[XXX]
- 验证方法：[我要做什么来验证]
- 预期结果：[如果假设正确会看到什么]
- 备选方向：[如果验证失败，下一个要排查的是什么]
```

**如果假设不明确，回到 Step 2 继续研究。**

---

### Step 4: [ACT] 精确修正

**触发条件：** 有明确假设且知道验证方法后。

**铁律：**
- **一次只改一个变量。** 不要同时改配置、换工具、改参数——你不知道哪个有效。
- **用最小操作验证假设。** 能用一个 curl 验证的，不要改配置文件。
- **修正后立即验证。** 不等待、不推测、不看"好像好了"——用明确的验证标准判断。

**输出模板：**
```
【执行】
- 修正操作：[具体做了什么]
- 验证方法：[怎么验证的]
- 实际结果：[看到了什么]
- 符合预期：[是/否]
```

**如果不符合预期 → 回到 Step 1。** 这意味着之前的假设有误，需要重新诊断。

---

### Step 5: [LEARN] 归纳记录

**触发条件：** 问题解决后。

**强制行为（简短）：**

- 根因是什么？（一句话）
- 为什么会犯这个错误？（反省视角）
- 下次遇到类似症状，什么信号能让我更快定位？

---

## 铁律总结

| # | 铁律 | 说明 |
|---|------|------|
| 1 | 失败后禁止直接重试 | 先完成 Step 1，否则所有重试都是浪费 |
| 2 | 不确定根因禁止凭猜测行动 | 必须进入 Step 2 研究 |
| 3 | 同一方向失败 3 次强制暂停 | 激活全局诊断：这些失败的共同路径是什么？ |
| 4 | 请求用户操作必须有依据 | 能说清为什么是必要的，排除已做了哪些诊断 |
| 5 | 一次只改一个变量 | 多个变更混在一起，你不知道哪个有效 |

---

## 禁止行为清单

- ❌ "让我试试另一个方法"（没诊断为什么上一个方法失败）
- ❌ "换一个工具试试"（没排查当前工具为什么不行）
- ❌ "你试试重启/再点一次"（我在猜，在让用户替我验证猜测）
- ❌ 连续 3 条命令都失败还在发第 4 条
- ❌ 从源码/错误提示里翻出一个没见过的环境变量就用
- ❌ 同时改多个配置项然后期望"其中某个应该能解决"

---

## 行为转变对比

| 场景 | ❌ 旧行为 | ✅ 新行为 |
|------|----------|----------|
| 命令报错 | 换个参数重试 | 先读完错误信息，提取关键线索 |
| 工具连不上 | 换另一个工具 | 诊断为什么连不上：端口？权限？版本？ |
| 不确定怎么用 | grep 源码找参数 | 读官方文档和 help 输出 |
| 多次失败 | 继续尝试更多方案 | 暂停，归纳三个失败案例的共同路径 |
| 想让用户帮忙 | "你重启一下" | "我已诊断出[依据]，需要你执行[X]来验证" |

