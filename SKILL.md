# Stop-Research-Act: 目标导向的问题解决 SOP

## The Iron Law

```
1. 始终记住目标。偏离目标是最大的失败。
2. 失败后，不诊断就重试 = 浪费。
3. 不确定时，先搜索再行动 = 不猜。
```

---

## 完整循环

```
【第零步: PLAN】制定计划 → 执行
     ↓ 失败
【第一步: STOP】理解失败（分类 A/B/C）
     ↓
【第二步: RESEARCH】搜索文档/社区
     ↓
【第三步: REVISE】修正假设，更新计划
     ↓
【第四步: EXECUTE】执行修正
     ↓
【第五步: CHECK】验证结果 + 是否偏离目标
     ↓
回到执行（或重新 PLAN）
```

---

## 第零步: [PLAN] 制定计划

**在动手之前，先制定一个明确的计划：**

1. **明确目标**（一句话说清要达成什么）
2. **分解步骤**（每一步做什么，预期结果是什么）
3. **选择工具/方案**（用哪个工具，为什么是这个工具，依据是什么）
4. **设定成功标准**（怎么知道做完了？做完后用户能看到什么？）

**输出：**
```
【计划】
- 目标：[一句话]
- 方案：[用什么工具/方法，依据文档/已验证路径]
- 步骤：1→2→3
- 成功标准：[用户最终能看到什么]
```

---

## 第一步: [STOP] 诊断失败

**遇到任何失败 → 先诊断，不重试。**

1. 完整读完错误信息
2. 分类：A=用法错 / B=环境问题 / C=系统行为变化
3. 用最原始方式验证依赖链（curl/ls/lsof）

**输出：**
```
【诊断】
- 错误：[关键信息]
- 分类：[A/B/C]
- 根因假设：[如果我能解释错误原因的话]
```

---

## 第二步: [RESEARCH] 搜索研究

**根因不确定 → 先查文档/搜索，禁止猜测。**

1. 官方文档 → GitHub Issues → 社区
2. 这个工具的正确用法是什么？我的用法对吗？

**输出：**
```
【研究】
- 来源：[文档/Issue URL]
- 结论：[官方/社区怎么说]
```

---

## 第三步: [REVISE] 修正计划

**研究完成后，更新计划：**

1. 我的假设是什么？
2. 新的执行步骤是什么？（一次只改一个变量）
3. 是否偏离了最初的目标？（强制目标检查）

**输出：**
```
【修正计划】
- 假设：[根因是XXX]
- 修正方案：[最小化修改，一次改一个变量]
- 目标检查：[✅ 仍在朝着原始目标前进 / ❌ 偏离了，需要纠正]
```

---

## 第四步: [EXECUTE] 执行修正

- 一次只改一个变量
- 立即验证
- 记录结果

---

## 第五步: [CHECK] 验证 + 目标对齐

**修正后必须问两个问题：**

1. 修正成功了吗？（假设验证通过了吗？）
2. **我还在朝着原始目标前进吗？**（目标对齐检查）

**如果连续 3 次同一方向失败 → 强制暂停，归纳共同路径，重新搜索研究。**

---

## 铁律（5 条）

| # | 铁律 |
|---|------|
| 1 | **始终记住目标，每次修正后检查是否偏离** |
| 2 | 失败后禁止不诊断就重试 |
| 3 | 不确定根因必须先搜索，禁止猜测 |
| 4 | 同一方向失败 3 次，强制暂停并归纳 |
| 5 | 一次只改一个变量 |

---

## 快速自检（每次操作后）

```
1. 我的上一次操作是：诊断 / 试探 / 修复？
2. 如果是试探 → 违反了规则 2，回到 STOP
3. 我的目标还在吗？我偏离了吗？
```
